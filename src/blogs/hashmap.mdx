# Modern Hashmaps

Recently, I've been looking into how high performance hashmaps work.
As part of that exploration, I've tried to understand the state of the art.
In this post, we'll look at how Swiss Tables compare to Boost Hashmaps, and my own experimental new hashmap design.

I've chosen to code in Zig because its focus on performance and low-level control makes it easier to design high-performance data structures.
Additionally, the current Zig hashmap implementation has some performance [issues](https://github.com/ziglang/zig/issues/17851) that we'll try to improve upon.

We're going to look here at a hashmap with the following simple API:

```zig
fn compare(a: KeyType, b: KeyType) bool { ... }
fn hash(key: KeyType) u64 { ... }

const HashMap = struct {
    pub fn init(allocator: *std.mem.Allocator) HashMap { ... }
    pub fn insert(self: *HashMap, key: KeyType, value: u64) void { ... }
    pub fn get(self: *HashMap, key: KeyType) ?u64 { ... }
    pub fn remove(self: *HashMap, key: KeyType) void { ... }
}
```

To use this HashMap, we use 64 bit values because they pack nicely, and can represent an arbitrary pointer to memory.

## The Problem Of Clustering

<ProbingDemo />

As the diagrams run, notice how the average chain length of linear probing is noticeably higher than triangular probing.

This is caused by [clustering](https://en.wikipedia.org/wiki/Primary_clustering), because the longer the chain, the more likely it is that new keys will hash to the same index, causing the chain to grow even longer.

Even though quadratic probing reduces clustering, it still has some locality preference, which leads to similar clustering.

Note that we could avoid this completely with rehashing, but that would be expensive (both to rehash, and because of the unpredictable memory accesses).

# Robin Hood Hashmaps

One slight variant of the standard hashmap is to include the distance from the ideal bucket in the hash table, and to use that distance to determine which key to evict when a collision occurs.
This is called Robin Hood Hashing, and it has the advantage of reducing the variance in lookup times, because it tries to keep all keys as close to their ideal home as possible.
One additional trick is that robin hood hashing can avoid tombstones, because when a key is removed, we know which keys belong to which runs and can therefore shift them back into the correct location.

# Swiss Tables

Swiss Tables are a modern SIMD friendly variation on quadratic probing with tombstones.
They store large buckets of 16 entries, and use a 7 bit fingerprint to determine if a key is present in the bucket.
Because SIMD instructions can operate on 16 bytes at a time, this allows for very fast lookups a bucket at a time (which drastically reduces the number of buckets that need to be searched).


```zig
const tags: @Vector(16, u8) = self.loadTagGroup(bucket);

const tag_lane: @Vector(16, u8) = @splat(tag);
const matches: @Vector(16, u8) = tags == tag_lane;
```

# Cuckoo Filters

We can take the bucketing idea of swiss tables and combine it with the classic cuckoo filters (but this time store actual data) idea to make a SIMD cuckoo hashtable.

In essence, Cuckoo filters move the work from the lookup phase to the insertion phase, where we must find careful buckets for the key.

Unsurprisingly, because Cuckoo filters require only two SIMD block lookups for each key, they have extremely fast lookups (constant time).
Similarly, they have extremely fast deletions, because we can just remove the key from the bucket.

They push quite a bit more work to inserts, where they are forced to move keys around to make room, leading to potentially long "kick" chains.

One thing to note is that all the fast designs actually use this cuckoo bucketing idea, where we can compare and insert into anywhere in an entire bucket.
